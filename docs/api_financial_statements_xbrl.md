# [OpenDART] 재무제표 원본파일(XBRL) API 개발 가이드

이 문서는 OpenDART에서 제공하는 **재무제표 원본파일(XBRL)** API에 대한 상세 개발 가이드입니다. 이 API는 기업이 DART 시스템을 통해 제출한 국제 표준 재무 보고 형식인 XBRL 파일을 원본 그대로 다운로드할 때 사용됩니다.

## 1. 기본 정보

| 항목 | 상세 내용 |
| :--- | :--- |
| **메서드** | GET |
| **요청 URL** | `https://opendart.fss.or.kr/api/fnlttXbrl.xml` |
| **인코딩** | UTF-8 |
| **출력 포맷** | Zip FILE (binary) |

> [!NOTE]
> 이 API는 `고유번호` API와 마찬가지로 **ZIP 파일**로 결과를 제공합니다. 이 압축 파일 내에는 `.xbrl` 파일이 포함되어 있어, 회계 전문 지식이나 이를 분석할 수 있는 라이브러리가 필요할 수 있습니다.

---

## 2. 요청 인자 (Parameters)

| 요청키 | 명칭 | 타입 | 필수여부 | 값 설명 |
| :--- | :--- | :--- | :---: | :--- |
| `crtfc_key` | API 인증키 | STRING(40) | Y | 사용자가 발급받은 40자리의 인증키 |
| `rcept_no` | 접수번호 | STRING(14) | Y | 조회하고자 하는 공시 보고서의 접수번호 (14자리)<br>※ 공시검색 API 등에서 추출 |
| `reprt_code` | 보고서 코드 | STRING(5) | Y | 1분기: `11013`, 반기: `11012`, 3분기: `11014`, 사업보고서: `11011` |

---

## 3. 응답 결과 (Response Fields)

응답은 바이너리 형태의 ZIP 파일이며, 성공 시 파일 내에는 다음과 같은 파일들이 포함될 수 있습니다.
- `*.xbrl`: 재무제표 본문 (숫자 데이터 및 택소노미 참조)
- `*_label.xml`: 계정명 라벨 정보
- `*_cal.xml`: 계산식 정보
- `*_pre.xml`: 표시 형식 정보

| 응답키 | 명칭 | 출력 설명 |
| :--- | :--- | :--- |
| `status` | 에러 코드 | 정상 시 `000` |
| `message` | 안내 메시지 | 성공 시 "정상", 실패 시 에러 사유 |

---

## 4. 메시지 설명 (Status Codes)

주요 코드:
- `000`: 정상
- `013`: 조회된 데이터가 없음 (해당 보고서가 XBRL로 제출되지 않았거나 기간 만료)
- `014`: 파일이 존재하지 않음

---

## 5. 개발 참고 사항 및 활용 팁

### 5.1 XBRL(eXtensible Business Reporting Language) 이해
XBRL은 비즈니스 리포팅을 위한 XML 기반의 표준 언어입니다. 단순히 텍스트나 숫자의 나열이 아니라, 각 숫자 데이터에 '꼬리표(Tag)'를 붙여 기계가 읽을 수 있도록 만든 것입니다. 이를 통해 서로 다른 국가나 기업 간의 재무 정보를 같은 잣대로 쉽고 정확하게 비교할 수 있습니다.

### 5.2 왜 XBRL API를 사용하는가?
일반적인 `주요계정 API`는 이미 규격화된 몇 가지 항목만 보여주지만, **재무제표의 모든 항목과 주석 내의 표 정보까지 정밀하게 수집**하려면 XBRL 원본 파싱이 필수적입니다. 데이터 기반 주식 분석 시스템이나 퀀트 분석 서비스를 구축하는 경우 이 API를 핵심 원천 데이터로 사용합니다.

### 5.3 분석 도구 추천
XBRL은 구조가 매우 복잡하므로 직접 XML 파서를 만들기보다는 검증된 오픈소스 라이브러리를 활용하십시오.
- **Python**: `python-xbrl`, `arelle` (매우 강력한 분석 도구)
- **R**: `XRBL` 패키지

### 5.4 수집 프로세스 시나리오
1. **보고서 검색**: 요청일 기준 업로드된 새로운 공시 리스트를 `공시검색 API`로 가져옵니다.
2. **대상 필터링**: 정기보고서(`11011`, `11012` 등)만 골라내어 `rcept_no` 리스트를 만듭니다.
3. **원본 다운로드**: 본 API를 사용하여 개별 보고서의 XBRL ZIP 파일을 서버로 가져옵니다.
4. **파싱 및 적재**: `Arelle` 등을 사용하여 필요한 계정 과목 및 금액 정보를 추출하여 시계열 데이터베이스에 저장합니다.

### 5.5 한계점 및 주의사항
- **금융업종 제외**: 은행, 보험, 증권 등 금융업종은 일반 제조/서비스업과 재무제표 구조(택소노미)가 달라 XBRL로 제공되지 않거나 파싱이 매우 어렵습니다.
- **연결 재무제표**: 연결 재무제표와 별도 재무제표 데이터가 한 파일에 섞여 있는 경우가 많으므로 `Context` 정보를 정확히 구분하여 읽어야 합니다.
- **임시 가망성**: 기업이 제출한 보고서가 나중에 정정되는 경우 접수번호(`rcept_no`)가 변경될 수 있으므로, 항상 최신 정정 보고서가 있는지 확인하는 로직이 동반되어야 합니다.
- **사용자 키 관리**: 파일 다운로드는 네트워크 사용량이 크므로, 비효율적인 반복 요청을 자제하고 서버 사이드에서 스토리지를 관리하는 것이 경제적입니다.
